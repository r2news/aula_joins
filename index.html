PARTE 1

Por Que Dados Ficam Separados?

Princípios de Banco de Dados Relacional

Em um banco de dados relacional, cada tabela deve representar um assunto específico. Essa organização oferece vários benefícios:

✓ Evita Repetição

Não repetir o nome do cliente em cada venda. Armazena apenas uma vez.

⚡ Melhora Performance

Tabelas menores são processadas mais rapidamente pelo MySQL.

🔧 Facilita Manutenção

Alterar dados centralizados é mais seguro e eficiente.

🛡️ Reduz Erros

Dados únicos em um lugar = menos inconsistências.

Exemplo: Sistema de Vendas

Tabela Responsabilidade
clientes Armazena cadastro: id, nome, cidade
vendas Armazena transações: id_venda, id_cliente, data, valor
PARTE 1

Chave Primária e Chave Estrangeira

🔑 O Fundamento do Relacionamento

Antes de usar JOIN, você precisa entender como as tabelas se conectam.

Chave Primária (PK)

Definição: Identificador único de cada registro em uma tabela.

Função: Garante que cada linha seja única (como um CPF).

Exemplo:
id_cliente INT PRIMARY KEY

Chave Estrangeira (FK)

Definição: Referência a uma chave primária de outra tabela.

Função: Cria o vínculo entre duas tabelas.

Exemplo:
FOREIGN KEY (id_cliente)

Visualização da Relação

Tabela CLIENTES
id_cliente = 1 (PK)
←→
Tabela VENDAS
id_cliente = 1 (FK)

A chave estrangeira em VENDAS aponta para a chave primária em CLIENTES

PARTE 1

Preparando o Ambiente

Criar Banco de Dados e Tabelas

-- Criar o banco de dados
CREATE DATABASE aula_joins;
USE aula_joins;

-- Criar tabela de clientes
CREATE TABLE clientes (
    id_cliente INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    cidade VARCHAR(100)
);

-- Criar tabela de vendas com chave estrangeira
CREATE TABLE vendas (
    id_venda INT PRIMARY KEY AUTO_INCREMENT,
    id_cliente INT,
    data_venda DATE,
    valor DECIMAL(10,2),
    FOREIGN KEY (id_cliente) REFERENCES clientes(id_cliente)
);
⚙️ Explicação da Estrutura:
• AUTO_INCREMENT: Gera IDs automaticamente
• NOT NULL: Campo obrigatório
• FOREIGN KEY: Garante que o id_cliente em VENDAS existe em CLIENTES

PARTE 1

INNER JOIN - Conceito

O Tipo de JOIN Mais Comum

INNER JOIN mostra apenas os registros que possuem correspondência em ambas as tabelas.

Visualização Lógica

TABELA A
(clientes)
∩
TABELA B
(vendas)

Retorna apenas a interseção entre as tabelas

Aspecto INNER JOIN
O que mostra? Apenas registros com correspondência em ambas as tabelas
Usa qual coluna? A chave primária e a chave estrangeira (ON)
NULL aparece? Nunca - registros sem match são excluídos
Caso de uso Quando você quer apenas registros vinculados
PARTE 1

INNER JOIN - Exemplo Prático

SELECT 
    clientes.nome,
    clientes.cidade,
    vendas.data_venda,
    vendas.valor
FROM clientes
INNER JOIN vendas
    ON clientes.id_cliente = vendas.id_cliente;
📍 Linha ON:
ON clientes.id_cliente = vendas.id_cliente

Esta linha diz ao MySQL: "Junte as tabelas quando o id_cliente da tabela clientes for igual ao id_cliente da tabela vendas"
nome cidade data_venda valor

João Souza Contagem 2026-06-03 850.00
Maria Silva Ibirité 2026-06-10 2300.00
Ana Costa Belo Horizonte 2026-06-15 1200.00
⚠️ Observe: Carlos Lima (id=4) não aparece no resultado! Ele está na tabela CLIENTES, mas não tem nenhuma venda em VENDAS. No INNER JOIN, registros sem correspondência são excluídos.

Alias: Escrita Profissional SQL

Utilizando Apelidos para Tabelas e Colunas

Em consultas profissionais, usamos alias (apelidos) para deixar o código mais limpo e organizado.

SELECT 
    c.nome,                    -- alias 'c' para clientes
    c.cidade,
    v.data_venda,             -- alias 'v' para vendas
    v.valor
FROM clientes c                -- Define alias 'c' aqui
INNER JOIN vendas v            -- Define alias 'v' aqui
    ON c.id_cliente = v.id_cliente;

Sem Alias (Ruim)

clientes.nome

clientes.cidade

vendas.data_venda

Repetitivo e cansativo

Com Alias (Bom)

c.nome

c.cidade

v.data_venda

Conciso e profissional

✅ Boas Práticas:
• Use abreviações curtas e lógicas (c para clientes, v para vendas)
• Mantenha consistência na mesma consulta
• Qualifique sempre as colunas ambíguas

LEFT JOIN - Conceito

Quando Você Quer Manter Todos da Tabela Principal

LEFT JOIN mostra todos os registros da tabela da esquerda, completando com dados da tabela da direita quando houver correspondência.

Visualização Lógica

TABELA A
(LEFT)
→
TABELA B
(RIGHT)

Retorna tudo da esquerda + dados correspondentes da direita

Aspecto INNER JOIN LEFT JOIN
Registros? Apenas com match Todos da esquerda
Sem correspondência? Excluído Mantém com NULL
Melhor para? Dados vinculados Relatórios completos

    ON c.id_cliente = v.id_cliente;

PARTE 2

Encontrando Registros Sem Venda

Uma Aplicação Importante do LEFT JOIN

O LEFT JOIN é perfeito para encontrar registros que existem na tabela principal, mas não têm correspondência na tabela relacionada.

SELECT 
    c.nome,
    c.cidade
FROM clientes c
LEFT JOIN vendas v
    ON c.id_cliente = v.id_cliente
WHERE v.id_venda IS NULL;
nome cidade
Carlos Lima Ibirité
🔍 Como funciona?
1. LEFT JOIN mantém TODOS os clientes
2. WHERE v.id_venda IS NULL filtra apenas os que não têm venda
3. Resultado: clientes sem histórico de vendas

Casos de Uso Reais

✓ Clientes cadastrados que nunca compraram (alvo de marketing)

✓ Produtos cadastrados mas não vendidos (revisar estoque)

✓ Alunos matriculados sem notas lançadas (verificar professores)

✓ Equipamentos cadastrados sem chamados (verificar utilização)

PARTE 2

RIGHT JOIN - Conhecimento Básico

O Inverso do LEFT JOIN

RIGHT JOIN mostra todos os registros da tabela da direita, mantendo NULL quando não há correspondência na tabela da esquerda.

SELECT 
    c.nome,
    c.cidade,
    v.data_venda,
    v.valor
FROM clientes c
RIGHT JOIN vendas v
    ON c.id_cliente = v.id_cliente;
💡 Dica Profissional: Na prática, você raramente precisa de RIGHT JOIN. A mesma lógica pode ser feita com LEFT JOIN invertendo a ordem das tabelas: 
-- Equivalente a RIGHT JOIN:
SELECT 
    c.nome,
    c.cidade,
    v.data_venda,
    v.valor
FROM vendas v
LEFT JOIN clientes c
    ON v.id_cliente = c.id_cliente;
📌 Recomendação: Domine bem INNER JOIN e LEFT JOIN. Eles resolvem 95% dos casos. RIGHT JOIN é mais raro em trabalhos iniciais.
PARTE 2

Filtros (WHERE) em JOINs

Combinando JOIN com Condições

Você pode usar WHERE para filtrar os resultados de uma consulta com JOIN, selecionando apenas dados que atendem a critérios específicos.

SELECT 
    c.nome,
    c.cidade,
    v.data_venda,
    v.valor
FROM clientes c
INNER JOIN vendas v
    ON c.id_cliente = v.id_cliente
WHERE c.cidade = 'Ibirité';
nome cidade data_venda valor
Maria Silva Ibirité 2026-06-01 1500.00
Maria Silva Ibirité 2026-06-10 2300.00
🔗 Ordem de Execução:
1. FROM/JOIN une as tabelas
2. ON define o relacionamento
3. WHERE filtra os resultados (último)

Ordenação e Agrupamento

ORDER BY: Organizando Resultados

SELECT 
    c.nome AS cliente,
    v.data_venda,
    v.valor
FROM clientes c
INNER JOIN vendas v
    ON c.id_cliente = v.id_cliente
ORDER BY v.valor DESC;
cliente data_venda valor
Maria Silva 2026-06-10 2300.00
Maria Silva 2026-06-01 1500.00
Ana Costa 2026-06-15 1200.00
João Souza 2026-06-03 850.00

GROUP BY: Agrupando e Agregando Dados

SELECT 
    c.nome AS cliente,
    COUNT(v.id_venda) AS quantidade_vendas,
    SUM(v.valor) AS total_vendido
FROM clientes c
INNER JOIN vendas v
    ON c.id_cliente = v.id_cliente
GROUP BY c.nome;
cliente quantidade_vendas total_vendido
Maria Silva 2 3800.00
João Souza 1 850.00
Ana Costa 1 1200.00
PARTE 2

GROUP BY com LEFT JOIN

Incluindo Clientes Sem Vendas nos Totais

Quando combinamos LEFT JOIN com GROUP BY, conseguimos gerar relatórios que incluem registros sem correspondência. Para evitar NULL nas somas, usamos IFNULL.

SELECT 
    c.nome AS cliente,
    COUNT(v.id_venda) AS quantidade_vendas,
    IFNULL(SUM(v.valor), 0) AS total_vendido
FROM clientes c
LEFT JOIN vendas v
    ON c.id_cliente = v.id_cliente
GROUP BY c.nome;
cliente quantidade_vendas total_vendido
Maria Silva 2 3800.00
João Souza 1 850.00
Ana Costa 1 1200.00
Carlos Lima 0 0.00
✨ IFNULL: Converte NULL em 0 para somas. Deixa o relatório limpo e compreensível.
PARTE 2

Erros Comuns e Como Evitar

Erro Por que acontece? Como corrigir
Esquecer ON MySQL não sabe como ligar as tabelas Sempre declare a condição após JOIN
Coluna errada no ON Consulta retorna dados incorretos Relacione PK com FK (chave primária com estrangeira)
Não usar alias Código fica grande e ilegível Use c para clientes, v para vendas
Confundir INNER/LEFT Resultado exclui registros importantes INNER = com match; LEFT = todos da esquerda
Não tratar NULL Relatórios com campos vazios Use IFNULL, COALESCE, ou IS NULL
🚨 Erro Clássico: Usar INNER JOIN quando você precisava de LEFT JOIN. Seu relatório aparenta estar "incompleto" porque está excluindo clientes sem vendas.

Próximo Conteúdo: VIEW

Por Que aprender VIEW?

Uma VIEW é uma consulta SQL salva no banco de dados. Depois que você domina JOIN, fica fácil aprender VIEW, pois muitas VIEWs armazenam consultas com JOIN!

💡 Vantagem: Em vez de escrever a mesma consulta com JOIN toda vez, você cria uma VIEW e a reutiliza em relatórios, dashboards e sistemas. 
-- Criar uma VIEW que armazena uma consulta com JOIN
CREATE VIEW vw_vendas_clientes AS
SELECT 
    v.id_venda,
    c.nome AS cliente,
    c.cidade,
    v.data_venda,
    v.valor
FROM vendas v
INNER JOIN clientes c
    ON v.id_cliente = c.id_cliente;

-- Depois, usar a VIEW é simples:
SELECT * FROM vw_vendas_clientes;
✅ Resultado: Uma VIEW que combina dados de vendas com informações de clientes. Profissional, reutilizável e eficiente!
